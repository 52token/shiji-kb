# SKILL 10b: 每日工作日志维护 — 项目记忆的守护者

> **核心理念**：日志是项目的记忆，微信通知是团队的脉搏，改动意义是价值的明镜。

---

## 一、工作日划分规则

### 1.1 时间分界线：早上7:00

**本系统以早上7:00为工作日分界线，而非传统的午夜0:00**

### 1.2 时间范围定义

- `YYYY-MM-DD.md` 包含的提交时间：`(YYYY-MM-DD) 07:00` 至 `(YYYY-MM-DD+1) 07:00`
- 例如：`2026-03-23.md` 包含 `2026-03-23 07:00` ~ `2026-03-24 07:00` 的所有提交

### 1.3 设计原因

1. **符合实际作息** - 凌晨0-7点的工作通常是前一天的延续
2. **方便统计** - 次日早上7点后可以完整统计前一天的工作
3. **避免切分** - 深夜加班不会被切分到两天

### 1.4 时间范围示例

| 提交时间 | 计入日志 | 说明 |
|---------|---------|------|
| 2026-03-23 08:00 | 2026-03-23.md | 白天工作算当日 |
| 2026-03-23 23:30 | 2026-03-23.md | 深夜工作算当日 |
| 2026-03-24 02:00 | 2026-03-23.md | 凌晨算前一日 |
| 2026-03-24 06:59 | 2026-03-23.md | 7点前算前一日 |
| 2026-03-24 07:00 | 2026-03-24.md | 7点起算次日 |

## 二、详细日志生成

### 2.1 日志生成脚本

```bash
# 日志文件位置
logs/daily/YYYY-MM-DD.md

# 生成详细日志
cd logs/daily
python generate_log.py YYYY-MM-DD

# 示例
python generate_log.py 2026-03-17
```

### 2.2 详细日志格式

脚本自动生成的日志格式：

```markdown
# 工作日志 YYYY-MM-DD

## 核心功能变化

### 分类1
- 变更描述1
- 变更描述2

### 分类2
- 变更描述1

## 技术细节
- 提交次数: N
- 涉及文件: M个
```

## 三、微信群通知生成

### 3.1 工作流程

1. **读取详细日志** - 使用Read工具读取 `logs/daily/YYYY-MM-DD.md`
2. **检查提交次数** - 判断是否有提交
   - 有提交 → 执行3.2节（常规通知）
   - 无提交 → 执行3.3节（太史公曰）
3. **提取核心工作** - 从日志中提取关键成果
4. **生成微信通知** - 按格式要求生成纯文本摘要
5. **更新日志文件** - 将通知追加到日志最前面

### 3.2 有提交日的通知格式

**格式要求**：

```
【史记知识库 YYYY-MM-DD】

· 核心工作1（包含关键数字）
· 核心工作2（包含关键数字）
· 核心工作3
...
· 提交N次代码
```

**具体要求**：

1. **列表格式**
   - 使用全角中点（·）作为列表符号
   - 每行一个工作项
   - 最多10项（包括"提交N次"）
   - 一般6-8项为宜

2. **纯文本格式**
   - 无markdown语法
   - 无阿拉伯数字编号
   - 无多级结构
   - 不使用emoji（除标题的【】）

3. **语言风格**
   - 平实易懂的中文
   - 避免技术黑话
   - 简化版本号（如v2.8→直接说做了什么）
   - 将技术动作转化为成果描述

4. **内容选择**
   - 提取5-9条核心工作
   - 每条尽量包含关键数字
   - 最后一项固定为"提交N次代码"

5. **优先级**
   - 带数字的成果（如"修正615处"）优先
   - 完成性工作（如"完成XX"）优先
   - 新增功能优先于维护
   - 核心模块优先于辅助工具

**技术术语简化示例**：

| 原文 | 简化后 |
|------|--------|
| v2.8格式统一：18类实体全部迁移为〖TYPE X〗格式 | 统一18类实体标注格式 |
| 内联消歧语法 v2.7：扩展到所有13种实体类型 + lint修复 | 消歧语法扩展到13类实体 |
| 重建HTML/实体索引，更新反思文档和SKILL规则 | 更新实体索引和反思文档 |
| 人名实体跨章反思：615处修正（邦国/氏族/动词/边界损坏） | 跨章反思修正615处人名标注 |

### 3.3 无提交日的太史公曰

**当日无提交时，使用太史公文风书写**：

**格式要求**：

```
【史记知识库 YYYY-MM-DD】

太史公曰：是日无所书。

<32字赞文，8个四字句>
```

**赞文规范**：

- **总字数**：严格32字（**仅计汉字，不含标点**）
- **句式**：必须全部使用**四字句**（每句恰好四个汉字）
- **结构**：8个四字句 = 32字
- **文风**：参考 `labs/sima-qian-style/SKILL.md` 的太史公曰风格
- **内容主题**：
  - 可赞项目建设之不易
  - 可赞持续积累之重要
  - 可赞休整思考之必要
  - 可叹工程浩大需时日
- **修辞手法**：
  - 使用对仗、排比
  - 简短有力，点到为止
  - 避免冗长说教
  - 尽可能押韵
- **禁止事项**：
  - ❌ 不能使用三字句（如"久也"）
  - ❌ 不能使用五字句（如"五十七万言"）
  - ❌ 不能使用六字句或更长
  - ❌ 不重复正文内容
  - ❌ 不使用现代白话（"的""了""很"等）
- **变化性**：
  - 每天的赞文必须不同
  - 可变换主题（休整/思考/持续/工程等）
  - 可变换句式（对仗/排比/递进等）

**示例1**（休整主题，32字）：
```
大业未成，日复一日；
张弛有度，蓄力待时。
善于休整，方能长久；
厚积薄发，终有所成。
```
（8句×4字 = 32字 ✅）

**示例2**（思考主题，32字）：
```
代码暂止，思虑不息；
磨刀不误，砍柴之工。
三思后行，智者所为；
谋定后动，方克有济。
```
（8句×4字 = 32字 ✅）

**示例3**（持续主题，32字）：
```
百卷宏业，非一日成；
偶有停歇，理之常然。
持之以恒，乃成大器；
锲而不舍，金石可镂。
```
（8句×4字 = 32字 ✅）

**示例4**（工程浩大，32字）：
```
五十万言，岂能速竟？
休整思谋，长远之计。
善始善终，必有所成；
积跬致远，不可须臾。
```
（8句×4字 = 32字 ✅）

**计数验证方法**：
```
示例1验证：

张弛有度（4字）

总计：8句×4字 = 32字 ✅
标点（不计入）：，；，。，；，。
```

## 四、改动意义分析

### 4.1 三个核心问题

**在完成微信通知总结后，必须再追加一段"改动意义"分析**，回答：

1. **为什么做这些事？** - 动机和背景
2. **要解决什么问题？** - 痛点和目标
3. **会影响什么最终交付物？** - 成果和价值

### 4.2 写作要求

- **独立章节**：使用 `## 改动意义` 作为二级标题
- **位置**：放在微信通知摘要之后，详细日志之前
- **长度**：150-200字（不超过微信手机阅读两屏）
- **结构**：分段回答三个问题，每个问题1-2句话
- **语言**：平实、直白，避免术语堆砌
- **视角**：从项目全局和长期价值考虑
- **简洁原则**：能一句话说清楚就不用两句话

### 4.3 示例（2026-03-28）

```markdown
## 改动意义

**为什么做这些事？**

工作日志和README文档滞后，03-25~03-27三天工作未记录，史源推理150K字新增案例未在索引中体现，影响项目可追溯性和可导航性。

**要解决什么问题？**

日志断档、无提交日处理规范缺失、文档导航失效（12个史源推理案例无索引）。

**会影响什么最终交付物？**

建立完整03-25~03-29时间线为CHANGELOG提供数据源，"太史公曰"规范成为可复用SKILL，4层README更新畅通导航链路。
```

**字数统计**：约110字（符合微信手机两屏阅读）

### 4.4 其他场景示例

**场景1：无提交日的改动意义**
```markdown
## 改动意义

**为什么没有提交？**

当日时间主要投入到其他工作事务中，未进行本项目的代码开发。

**对项目有什么影响？**

无实质影响。工作日志保持连续记录，用"太史公曰"幽默处理空白日。
```

**说明**：无提交日通常无需过度分析，简明扼要即可。"太史公曰"只是一个幽默的文化梗。

**场景2：修复大量标注错误**

```markdown
跨章反思发现615处人名标注错误，暴露单章反思局限性——邦国名、氏族名易被误标为人名。

建立跨章实体规范化机制，通过全局词频识别系统性误标模式。

**会影响什么最终交付物？**

提升知识图谱查准率，错误模式转化为规则防止重复，修正后产生更可靠的HTML索引。
```

```markdown
**要解决什么问题？**

开发 `lint_text_integrity.py` 自动识别空白段落和标注密度异常区域。

**会影响什么最终交付物？**

成为质量保障必查项，防止实体缺失，降低长期维护成本。
```

**场景4：文档重构**

```markdown
**为什么做这些事？**

项目从单一任务演变为多模块知识库，扁平化目录难以组织日益增长的内容。

**要解决什么问题？**

建立分层目录规范（skills/labs/resources/），每个子目录有独立README索引。

**会影响什么最终交付物？**

新加入者快速理解项目结构，SKILL文档分类清晰便于复用，新模块有明确归属避免碎片化。
```

## 五、完整实施流程

### 5.1 步骤总览

```
1. 生成详细日志 (generate_log.py)
   ↓
2. 读取日志文件 (Read工具)
   ↓
3. 检查是否有提交
   ├─ 有提交 → 生成常规微信通知
   └─ 无提交 → 生成太史公曰
   ↓
4. 撰写改动意义分析
   ↓
5. 更新日志文件（追加到最前面）
   ↓
6. 输出确认
```

### 5.2 详细步骤

**步骤1：生成详细日志**

```bash
cd logs/daily
python generate_log.py 2026-03-17
```

**步骤2：读取日志文件**

```python
Read logs/daily/2026-03-17.md
```

**步骤3：分析提取**

- 列出所有核心变化
- 标注优先级
- 提取关键数字
- 判断有无提交

**步骤4：生成总结**

有提交时：
- 按优先级选择5-9条
- 简化技术术语
- 加上提交次数

无提交时：
- 使用太史公曰格式
- 撰写32字赞文（8个四字句）

**步骤5：撰写改动意义分析**

基于当日工作内容，回答三个核心问题：
- **为什么做这些事？** - 从项目背景和动机出发
- **要解决什么问题？** - 具体痛点和目标
- **会影响什么最终交付物？** - 成果和长期价值

**写作要点**：
- 长度：150-200字
- 结构：三个问题分段回答
- 视角：项目全局和长期价值
- 语言：平实直白，避免术语堆砌

**步骤6：更新日志文件**

使用Edit工具，在日志文件的**最前面**插入两部分内容：

```markdown
# 工作日志 YYYY-MM-DD

## 微信群通知

```
【史记知识库 YYYY-MM-DD】

· 核心工作1（包含数字）
· 核心工作2（包含数字）
· 核心工作3
· 核心工作4
· 提交N次代码
```

---

## 改动意义

**为什么做这些事？**

（1-3句话说明背景和动机）

**要解决什么问题？**

（1-3句话说明痛点和目标）

**会影响什么最终交付物？**

（1-3句话说明成果和价值）

---

## 核心功能变化

（原有内容保持不变）
...
```

**重要**：
- 在标题 `# 工作日志 YYYY-MM-DD` 之后立即插入
- 使用 `## 微信群通知` 和 `## 改动意义` 作为二级标题
- 摘要用代码块包裹（方便复制）
- 改动意义使用markdown格式（不用代码块）
- 用 `---` 分隔线分隔各部分
- 不改动原有的详细内容

**步骤7：输出确认**

```
✅ 已将微信通知摘要和改动意义分析添加到 logs/daily/YYYY-MM-DD.md

可直接复制以下内容发送到微信群：

【史记知识库 YYYY-MM-DD】

· 核心内容1
· 核心内容2
· 核心内容3
· 提交N次代码
```

## 七、质量检查清单

### 7.1 微信通知检查

- [ ] 是否纯文本（无markdown语法）
- [ ] 是否无编号或层级结构
- [ ] 是否超过150字（应该≤150）
- [ ] 是否包含关键数字
- [ ] 是否易于非技术人员理解
- [ ] 是否适合直接复制到微信群

### 7.2 太史公曰检查（无提交日）

- [ ] 总字数是否为32字（仅计汉字）

- [ ] 是否共8句（8×4=32）

- [ ] 是否回答了三个核心问题（为什么/解决什么/影响什么）
- [ ] 长度是否在150-200字范围内
- [ ] 是否从项目全局视角而非技术细节出发
- [ ] 语言是否平实直白，避免术语堆砌
- [ ] 是否清晰说明了对最终交付物的影响
